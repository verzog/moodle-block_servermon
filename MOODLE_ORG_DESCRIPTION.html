A lightweight admin Dashboard block that gives Moodle site administrators a live view of server health — CPU, RAM, disk, top processes, Moodle page-performance metrics, cache store stats, session info, and historical metric logging with CSV export. All data is visible only to site administrators.

Resource gauges

Three colour-coded progress bars (green / amber / red at 60% / 80%) show at a glance:

  • CPU — aggregate % from the most recent scheduled-task snapshot (falling back to a live 500 ms /proc/stat two-sample delta), plus per-core breakdown bars and 1m/5m/15m load averages
  • RAM — used / free / total in GB from /proc/meminfo
  • Disk — used / free / total in GB for a configurable mount point (useful when Moodle data is on a separate partition such as /data)

Top processes by CPU

A collapsible panel that polls the server every 5 seconds via AJAX while open and stops when collapsed. Displays PID, process name, CPU%, MEM%, and the last CPU core the process ran on for the top 5 processes. Reads /proc/[pid]/stat directly with a 200 ms sample window; falls back to ps -A --sort=-%cpu -o pid,pcpu,pmem,psr,comm if /proc is unavailable. The Core column lets you cross-reference a saturated core in the per-core CPU bars with the process responsible.

Moodle debug footer — key metrics

A collapsible panel showing Moodle page-performance data for the current request:

  • Page load time, peak memory usage, DB read/write counts, total DB query time
  • Session handler type (file / Redis / Memcached / database), session size, and full Redis connection details when Redis sessions are active
  • MUC cache store performance — hits, misses, and I/O bytes broken down by static, application, session, and request caches
  • Advisory note if the application cache miss rate is high

OS users & PHP-FPM pools — shared-server isolation

A collapsible panel for auditing whether a multi-tenant server isolates sites correctly. It shows which OS user the current request runs as, lists login-capable operating-system users from /etc/passwd (UID ≥ 1000), and parses PHP-FPM pool configuration from the standard locations (/etc/php/*/fpm/pool.d/, /etc/php-fpm.d/, /usr/local/etc/php-fpm.d/). Each pool is rendered as a card showing its user/group, listen socket and hardening directives (open_basedir, chroot, listen.mode), cross-referenced against real accounts and flagged for problems such as generic/shared/root users, world-writable sockets, shared home directories or a missing open_basedir fence. It also detects whether /proc lacks hidepid (so other tenants' processes and command-line arguments are visible). A best-effort, always-unconfirmed verdict summarises the result.

Metric logging and CSV export

A scheduled task records CPU (per-core %), RAM%, and disk% every 5 minutes to a lightweight database table. A Download metrics CSV (last 7 days) link appears at the bottom of the block when data is available, with one column per CPU core.

Hosting type detection

A best-effort resource-scoring heuristic estimates the server environment (shared hosting, small VPS, or VPS/dedicated) from visible CPU cores, RAM, /proc readability, and the PHP process owner. The label is always marked as unconfirmed and the matched signals are listed so administrators can see why the guess was made.

Admin settings

A single setting under Site Administration → Plugins → Blocks → Server Monitor lets you specify the disk path to monitor (default /). This also applies to the scheduled task so logged values match what is displayed.

Requirements

  • Moodle 5.0 or higher
  • PHP 8.2 or higher
  • Linux server recommended (Windows Server supported with reduced functionality)

Security

Block content, the AJAX process endpoint, and the CSV export endpoint all require the moodle/site:config capability. No other user sees any output. No personal data is stored — the metric log records server-level values only.
