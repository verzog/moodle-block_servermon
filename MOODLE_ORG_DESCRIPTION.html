A lightweight admin Dashboard block that gives Moodle site administrators a live view of server health — CPU, RAM, disk, top processes, Moodle page-performance metrics, cache store stats, session info, and historical metric logging with CSV export. All data is visible only to site administrators.

Resource gauges

Three colour-coded progress bars (green / amber / red at 60% / 80%) show at a glance:

  • CPU — aggregate % from a 500 ms /proc/stat two-sample delta, plus per-core breakdown bars and 1m/5m/15m load averages
  • RAM — used / free / total in GB from /proc/meminfo
  • Disk — used / free / total in GB for a configurable mount point (useful when Moodle data is on a separate partition such as /data)

Top processes by CPU

A collapsible panel that polls the server every 5 seconds via AJAX while open and stops when collapsed. Displays PID, process name, CPU%, and MEM% for the top 5 processes. Reads /proc/[pid]/stat directly with a 200 ms sample window; falls back to ps aux if /proc is unavailable.

Moodle debug footer — key metrics

A collapsible panel showing Moodle page-performance data for the current request:

  • Page load time, peak memory usage, DB read/write counts, total DB query time
  • Session handler type (file / Redis / Memcached / database), session size, and full Redis connection details when Redis sessions are active
  • MUC cache store performance — hits, misses, and I/O bytes broken down by static, application, session, and request caches
  • Advisory note if the application cache miss rate is high

Metric logging and CSV export

A scheduled task records CPU (per-core %), RAM%, and disk% once per minute to a lightweight database table. A Download metrics CSV (last 7 days) link appears at the bottom of the block when data is available. The CSV includes a UTF-8 BOM for clean opening in Microsoft Excel, with one column per CPU core.

Hosting type detection

A best-effort heuristic identifies the server environment using six signal layers: control-panel fingerprints (cPanel, Plesk, DirectAdmin, Webmin, ISPConfig, HestiaCP, etc.), cloud provider detection via DMI vendor strings (AWS, Azure, GCP, DigitalOcean, Hetzner, Vultr, Linode, OVH), virtualisation type (KVM, VMware, Xen, Hyper-V, LXC/Docker), bare-metal chassis codes, OS fingerprint, and a resource-based score fallback.

Admin settings

A single setting under Site Administration → Plugins → Blocks → Server Monitor lets you specify the disk path to monitor (default /). This also applies to the scheduled task so logged values match what is displayed.

Requirements

  • Moodle 4.5 or higher
  • PHP 8.1 or higher
  • Linux server recommended (Windows Server supported with reduced functionality)

Security

Block content, the AJAX process endpoint, and the CSV export endpoint all require the moodle/site:config capability. No other user sees any output. No personal data is stored — the metric log records server-level values only.
